GIT

Workflow

A repository a.k.a. repo is nothing but a collection of source code.

There are four fundamental elements in the Git Workflow.

Working Directory, Staging Area, Local Repository and Remote Repository.

  1. It can be staged. Which means the files with the updated changes are marked to be committed to the local repository but not yet committed.
  2. It can be modified. Which means the files with the updated changes are not yet stored in the local repository.
  3. It can be committed. Which means that the changes you made to your file are safely stored in the local repository.

Exact-Space Standards

Any repo will by default be on the master branch. This branch will have the bug-free, tested code that goes into the production.

Any developments goes to the develop branch. Once the code is tested, it can be merged to the master by creating a pull request.

The develop branch should still be considered as a place where all the feature implementation after testing are pushed. Each feature implementation should be created on a separate feature branch which after testing can be merged to the develop branch which will finally be merged with the master.

The image represents the workflow followed in Exactspace. The master branch will be where the final release will be pushed.

The develop branch is where the major developments are pushed. Each feature development is pushed in a feature branch and finally merged with the develop branch.

Once the features are merged with the develop branch, a version release can be made by creating a release branch. A pull request is then created to merge with the master after reviewing. The code in master will then be deployed in the production server.

Hotfix are exceptional cases where we create a branch from master to make quick fixes and once tested will merge it directly with the master.

Project Boards

Our weekly sprints will now be moved to using project boards on Github.

A project item may be linked to a specific repo and an issue described in the repo.

Move tasks to different statuses as you work, and when it is pushed to testing, inform the relevant tester to validate and then this can be merged with develop branch with latest code.

Each issue marked in git should have sufficient information biased towards being able to test the output. Issues are also roadmap items, not necessarily bugs.

Points to note

Keep this in mind while working on any repo:

  • the feature branch name should be feature/<featurename>
  • create a pull request to the develop branch once the feature implementation is done. Do not merge directly to the master.
  • Best practice, assign a reviewer whenever you want to merge the code.
  • After merging the feature branch to develop, delete the feature branch both locally and in the remote.

Commands

Configurations

Configure your credentials to the current repository

git config user.name "Your Name Here"
git config user.email your@email.com

Configure credential globally

xxxxxxxxxx

Unset the configs

xxxxxxxxxx
git config --global --unset user.name

xxxxxxxxxx

Initializing Git

git clone

Every user “clones” a copy of a repository (a collection of files) and has the *full* history of the project on their own hard drive. This clone has all of the metadata of the original while the original itself is stored on a self-hosted server or a third party hosting service like GitHub. Locate to the directory you want to clone the repo. Copy the link of the repository you want and enter the following

xxxxxxxxxx
git clone remote_repository_URL
git pull

command to get files from the remote repository directly into the working directory. It is equivalent to a git fetch and a git merge.

xxxxxxxxxx
git pull origin master
git branch

Lets you know the current branch you are on.

xxxxxxxxxx
git branch
git checkout

Switch to a different branch

xxxxxxxxxx

Create a branch
xxxxxxxxxx
git checkout -b branchname

Contributing to the repo

git add

command to add a file that is in the working directory to the staging area.

xxxxxxxxxx
git add .

Note: "." specifies to add all the changes in the current directory. If you want to add specific file

xxxxxxxxxx

git status

Lists all new or modified files to be committed

xxxxxxxxxx

git commit

xxxxxxxxxx

git push

command to add all committed files in the local repository to the remote repository. So in the remote repository, all files and changes will be visible to anyone with access to the remote repository.

xxxxxxxxxx

git fetch AND git merge

When you use git pull, Git tries to automatically do your work for you. It is context sensitive, so Git will merge any pulled commits into the branch you are currently working in. git pull automatically merges the commits without letting you review them first.

When you git fetch, Git gathers any commits from the target branch that do not exist in your current branch and stores them in your local repository. However, it does not merge them with your current branch. This is particularly useful if you need to keep your repository up to date, but are working on something that might break if you update your files. To integrate the commits into your master branch, you use git merge.

Delete branch
xxxxxxxxxx
git branch -d branch_name
git branch -D branch_name

Note: The -d option is an alias for --delete, which only deletes the branch if it has already been fully merged in its upstream branch. You could also use -D, which is an alias for --delete --force, which deletes the branch "irrespective of its merged status." [Source: man git-branch]
