Deployment Guide

Application Configurations

An admin page to enter the configurations for the application had to be provided. Once this setup is done and the configurations are entered manually, app_config.py uses this data through APIs and serves it to other services.

Database Migration

Assuming migrating from dev machine(source) to production machine(target)

  1. Mongodump.sh: Takes UnitId to be backed up as input parameter. Run this script on the source server. A dump will be created under the folder /space/log/db_migration/<unitId> in a zip format.

    Execution command: ./mongodump.sh

    File location: /space/es-master/deploy/mongo-migration

  2. Move this zip file to the destination server.

  3. Mongorestore.sh: Takes UnitId from which data has to be migrated as input parameter. Unzips the dump folder and imports the data to the db

    Execution command: ./mongorestore.sh

    File location: /space/es-master/deploy/mongo-migration

Best Practice: Copy the dump to the shared mount folder between the source and the target server, and run the mongorestore.sh in the target server. This would remove the step of moving the dump to the target server.

Prerequisites:

Grant execute permissions to the scripts

chmod +x mongodump.sh

chmod +x mongorestore.sh

Mongo Migration for Demo

For the purpose of demo we create dummy units and reroute the actual live data to this dummy unit so as to project it to be live. To do this deployment we need a to create a dummy unit first using the loopback create unit API.

Assuming the /space/log directory is mounted between both the server. If not then the dump has to be moved manually to the target server.

Once unit is created follow the steps

  1. demo_mongodump.sh is executed on the source server. This takes the unitId to backup as input parameter and places the dump in /space/log/db_demo/<unitId>

  2. On the target machine, execute demo_mongorestore.sh. This takes unitId as parameter and fetches the dump, then replaces the unitId, dataTagId with the new unitId and dataTagId and then restores the dump to the database.

    Note: Currently the field to replace is hardcoded in the script. Also the service is meant to be run as a cron so any change in the live data will also be mirrored in the demo unit.

    File location: /space/es-master/deploy/mongo-migration

Services

  • Flow-er: This part is responsible for transforming rest api calls into mqtt publishes. Design to support multiple units.

    File location: $SRC/flow-er/index.5.py

    To allow publish for another unit - an if case needs to be added. (source check)

 

  • App-config: The central config for all realtime services and apis. A new unit is configured by adding the configurations in a separate block with that unitId as the key. A latest update is to make the configuration centralized in a place(database) and app_config reads from the db

    File location: $SRC/exsp-packages/app-config/app-config/app-config.py

 

  • Builds: There are 10 real-time software and 2 base images to be built. This is taken care by the script ./build-bs.sh.

    File location: /space/es-master/build.rns/

  • Launch-service: This is a responsible to start the realtime services. These realtime images must be declared in repositories.json

    File location: /space/es-master/deploy/launch-service

    Steps to edit repositories.json:

  1. Add the images to the repositories.json as shown in the below screenshot

  1. Followed by restarting launch-service

    Stop the service: /space/es-master/deploy/launch-service/kill5

    Start the service: /space/es-master/deploy/launch-service/run5

 

  • Starter: Starter is a place which generates instances of real-time service as services.json which is consumed by compose-gen which generates and starts the docker stack from it. To use it do ./startup.sh $unit_id (where $unit_id is the id of the new unit)
